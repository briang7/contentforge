# Custom Tools

While Claude Code ships with a powerful set of built-in tools, you can extend its capabilities by creating custom tools through MCP servers. Custom tools let you integrate proprietary APIs, internal systems, and domain-specific operations directly into Claude's workflow.

## Why Custom Tools?

Custom tools are valuable when you need Claude to:

- **Interact with internal APIs** (deployment systems, feature flag services, monitoring)
- **Access proprietary data sources** (internal databases, data warehouses, CRMs)
- **Perform domain-specific operations** (code generation from templates, schema validation)
- **Integrate with team workflows** (ticket management, CI/CD pipelines, documentation systems)

## Building a Custom Tool

Custom tools are implemented as MCP servers. Each server can expose one or more tools that Claude can discover and invoke.

### Setup

Create a new project for your MCP server:

```bash
mkdir my-mcp-tools && cd my-mcp-tools
npm init -y
npm install @modelcontextprotocol/sdk zod
npm install -D typescript @types/node
```

### Define Your Tool

Create a server with one or more tools:

```typescript
// src/server.ts
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({
  name: "deploy-tools",
  version: "1.0.0",
});

// Tool 1: Check deployment status
server.tool(
  "check_deploy_status",
  "Check the deployment status of a service",
  {
    service: z.string().describe("Service name (e.g., 'api', 'web', 'worker')"),
    environment: z.enum(["staging", "production"]).describe("Target environment"),
  },
  async ({ service, environment }) => {
    const response = await fetch(
      `https://deploy.internal.example.com/api/status/${environment}/${service}`,
      { headers: { Authorization: `Bearer ${process.env.DEPLOY_TOKEN}` } }
    );
    const status = await response.json();

    return {
      content: [
        {
          type: "text",
          text: `Service: ${service}\nEnvironment: ${environment}\nStatus: ${status.state}\nVersion: ${status.version}\nLast deployed: ${status.deployedAt}`,
        },
      ],
    };
  }
);

// Tool 2: Trigger a deployment
server.tool(
  "trigger_deploy",
  "Deploy a service to an environment",
  {
    service: z.string().describe("Service name"),
    environment: z.enum(["staging", "production"]).describe("Target environment"),
    version: z.string().describe("Git tag or commit SHA to deploy"),
  },
  async ({ service, environment, version }) => {
    const response = await fetch(
      `https://deploy.internal.example.com/api/deploy`,
      {
        method: "POST",
        headers: {
          "Content-Type": "application/json",
          Authorization: `Bearer ${process.env.DEPLOY_TOKEN}`,
        },
        body: JSON.stringify({ service, environment, version }),
      }
    );
    const result = await response.json();

    return {
      content: [
        {
          type: "text",
          text: `Deployment triggered.\nDeploy ID: ${result.id}\nTrack progress: ${result.url}`,
        },
      ],
    };
  }
);

// Connect via stdio transport
const transport = new StdioServerTransport();
await server.connect(transport);
```

### Tool Schema Best Practices

When defining tool parameters, use descriptive schemas:

```typescript
server.tool(
  "search_logs",
  "Search application logs by query and time range",
  {
    query: z.string()
      .describe("Log search query (supports Lucene syntax)"),
    service: z.string()
      .describe("Service name to search logs for"),
    timeRange: z.enum(["1h", "6h", "24h", "7d"])
      .default("1h")
      .describe("Time range to search"),
    level: z.enum(["debug", "info", "warn", "error"])
      .optional()
      .describe("Filter by log level"),
    limit: z.number()
      .min(1).max(100)
      .default(20)
      .describe("Maximum number of log entries to return"),
  },
  async (params) => {
    // Implementation here
  }
);
```

Key principles:
- Use `.describe()` on every parameter so Claude understands what to provide
- Set sensible defaults with `.default()`
- Use enums (`z.enum()`) when there is a fixed set of valid values
- Mark optional fields with `.optional()`
- Add validation constraints (`.min()`, `.max()`, `.regex()`)

## Registering Custom Tools

Add your MCP server to Claude Code's settings:

```json
// .claude/settings.json
{
  "mcpServers": {
    "deploy-tools": {
      "command": "node",
      "args": ["./tools/deploy-server/dist/server.js"],
      "env": {
        "DEPLOY_TOKEN": "${DEPLOY_TOKEN}"
      }
    }
  }
}
```

## Using Custom Tools

Once registered, your custom tools are available like any built-in tool:

```
> Check the deployment status of the API service in production

> Deploy the web service to staging with version v2.3.1

> Search the last 24 hours of API logs for "timeout" errors
```

Claude automatically selects the appropriate custom tool based on your request and the tool descriptions you provided.

## Testing Custom Tools

Test your MCP server independently before integrating with Claude Code:

```bash
# Run the server and send test input via stdin
echo '{"jsonrpc":"2.0","method":"tools/list","id":1}' | node dist/server.js
```

Or use the MCP Inspector tool:

```bash
npx @modelcontextprotocol/inspector node dist/server.js
```

This opens a web UI where you can discover and test your tools interactively.

## Permissions for Custom Tools

Custom MCP tools follow the same permission system as built-in tools. You can add them to your allowlist:

```json
{
  "permissions": {
    "allow": [
      "mcp__deploy-tools__check_deploy_status",
      "mcp__deploy-tools__trigger_deploy"
    ]
  }
}
```

The naming convention is `mcp__<server-name>__<tool-name>`.
